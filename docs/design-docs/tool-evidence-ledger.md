# Tool Evidence Ledger

The execution engine sends only bounded tool evidence into model reasoning. It never appends complete tool results or serialized prefixes to the prompt.

Producer projections pass through unchanged only after the execution engine independently verifies their typed envelope, exact tool identity, reported byte count, and 12 KiB ceiling. A failed producer check becomes a contract error rather than silently using generic fallback; dispatched writes retain an unknown outcome. Unknown non-producer results are recursively bounded by depth, object keys, array items, and UTF-8-safe string byte limits while remaining valid JSON with `_truncation` metadata. Tool arguments retained with evidence are separately limited so argument payloads cannot consume the run budget.

Before each model request, the engine regenerates one internally tagged `ACORNOPS_TOOL_EVIDENCE` message from a ledger capped at 48 KiB. The private tag is removed before the gateway request, but lets continuation logic distinguish synthetic evidence from identical user-authored text without elevating tool content to a system instruction. Both the run system prompt and evidence envelope identify tool fields, logs, and resource content as untrusted data whose embedded instructions must not override user intent, approvals, or safety rules. Semantic keys use the tool name plus operation ID, canonical resource identity, or a SHA-256 argument signature. New observations replace matching reads. Budget eviction removes unprotected observations first, then superseded protected observations, while retaining the latest error, write receipt, and `get_resource` verification. Omission counts survive approval continuation and are visible to the model.

Write arguments are validated against the advertised schema before approval. Invalid calls become protected `TOOL_ARGS_INVALID` evidence so the model can self-correct without presenting an unusable approval request. Union-schema failures report the closest concrete field errors instead of only a generic shape mismatch. Locally rejected calls still consume the total tool-call budget, but do not consume the identical-call safety budget because they were never eligible for approval or dispatch; valid identical calls remain bounded. Gateway validation remains authoritative before execution.

`patch_resource` has an additional provenance gate before approval or dispatch. Workload targets must exactly match the kind, namespace, name, UID, container identity, and current image from the latest successful Pod `get_resource` ownership projection already in the active ledger; the top-level target and ownership result must agree. A direct controller read cannot authorize a workload patch. Service and Ingress patches must match a direct resource projection. Missing, superseded, or mismatched evidence becomes `REMEDIATION_TARGET_NOT_RESOLVED`, so a model must return to the exact Pod instead of guessing a controller.

Successful image patches create a bounded verification requirement keyed by operation and target identity. A malformed success receipt that cannot create that requirement records `missing` immediately. A later fresh `get_resource` observation resolves a valid requirement as `verified` or `failed`; a run that terminates without matching evidence records `missing`, and cancellation records `cancelled`. Pending verification requirements survive approval continuation and expose only the bounded `tool` and `outcome` metric labels.

Any write error with `outcome: unknown` is terminal for that run, including transport, HTTP 5xx, and response-contract failures after dispatch. The engine tells the operator that the target must be inspected before retrying and does not return the uncertain receipt to the model for another attempt. This applies whether the uncertain result is loaded from continuation state, received during the current resume, or occurs on a write path configured without approval.

Approval execution state is authoritative during recovery. A previously completed approval reuses its stored receipt and is never dispatched again; an `executing` state observed by a second recovery attempt transitions to `unknown` and stops safely.

Eligible complete results are uploaded to the control plane before the compact `tool_call_completed` event is emitted. Approval continuation persists only the bounded gateway-provided model context, so approved AgentK writes retain their exact compact receipt without retaining arbitrary complete third-party output for the conversation lifetime. Complete eligible results use the separate seven-day artifact path. Ambiguous approved writes emit compact completion evidence before the run terminates. Upload failures set `artifactUnavailable` but do not fail the diagnosis.

Every durable `tool_call_completed` event is regenerated through the same bounded formatter, including local validation and repeat-limit errors. The formatter emits only the compact event allowlist, recomputes context size, and selects the fixed artifact metadata fields; complete results cannot enter the event stream through an alternate local path.

The normalized gateway response is read through a 5 MiB bounded stream. This is larger than the 2 MiB complete-result ceiling because generic MCP `content` may legitimately appear in both `full_result` and the pre-fallback `model_context`; crossing the envelope ceiling becomes `TOOL_RESULT_TOO_LARGE`, and a dispatched write is marked `outcome: unknown`.
